Admin Login with OTP Verification Authentication
Admin credentials are validated first. If correct, a 6-digit OTP is sent to the registered email. The JWT token is only issued after successful OTP verification.

Admin 2-Factor Login Flow

POST /auth/login → POST /auth/verify-login-otp
sequenceDiagram actor Admin participant LP as Login Page participant AC as Auth Controller participant UM as User Model participant ES as Email Service Admin->>LP: Enter email & password LP->>AC: POST /api/auth/login AC->>UM: findOne({ email }).select(+password) UM-->>AC: User document returned alt User not found or inactive AC-->>LP: 401 Invalid credentials LP-->>Admin: Show error message else Password mismatch AC->>UM: matchPassword(password) UM-->>AC: false AC-->>LP: 401 Invalid credentials LP-->>Admin: Show error message else Valid admin credentials AC->>UM: matchPassword(password) UM-->>AC: true Note over AC: role === 'admin' → OTP required AC->>UM: generateOTP('login') UM-->>AC: 6-digit OTP stored with 10-min expiry AC->>ES: sendOTPEmail(email, otp, 'login') ES-->>Admin: Email: "Your Login OTP is 123456" AC-->>LP: 200 { requiresOTP: true, email } LP->>LP: Switch to OTP verification screen LP-->>Admin: Show masked email + OTP input end Admin->>LP: Enter 6-digit OTP LP->>AC: POST /api/auth/verify-login-otp { email, otp } AC->>UM: findOne({ email }).select(+resetOTP +resetOTPExpire) UM-->>AC: User document alt OTP invalid or expired AC->>UM: checkOTP(otp) UM-->>AC: false AC-->>LP: 400 Invalid or expired OTP LP-->>Admin: Error shown, OTP field cleared else OTP valid AC->>UM: checkOTP(otp) UM-->>AC: true AC->>UM: Clear resetOTP & resetOTPExpire UM-->>AC: Saved AC->>UM: getSignedJwtToken() UM-->>AC: JWT token AC-->>LP: 200 { token, user data } LP->>LP: localStorage.setItem(token, user) LP-->>Admin: Redirect to Dashboard end

Create Bill (Billing / POS) Sales
Staff adds medicines to a bill. The system validates stock, deducts inventory using FIFO, calculates GST, and creates the bill record.

Create Bill Flow

POST /api/bills
sequenceDiagram actor Staff/Admin participant BP as Billing Page participant BC as Bill Controller participant MM as Medicine Model participant IM as Inventory Model participant BM as Bill Model participant CSM as CashierSession Model Staff/Admin->>BP: Enter customer details & scan/search medicines Staff/Admin->>BP: Set quantity, payment mode, discount BP->>BC: POST /api/bills { customerName, customerPhone, items, paymentMode, amountPaid } BC->>BC: validateBillPayload() — check required fields BC->>BC: Start MongoDB session & transaction loop For each bill item BC->>MM: findById(item.medicine) MM-->>BC: Medicine found with conversionFactor, GST info BC->>IM: Find ACTIVE batch (FIFO — earliest expiry first) IM-->>BC: Inventory batch with MRP, gstPercent BC->>BC: Calculate basePrice, CGST, SGST/IGST per unit BC->>IM: Deduct unitQuantity from quantityAvailable IM-->>BC: Inventory updated (status = EXHAUSTED if qty = 0) end BC->>BC: Sum subtotal, totalGst, apply discount → grandTotal BC->>CSM: findOne({ staff, status: OPEN }) CSM-->>BC: Active cashier session (or null) BC->>BM: Create Bill { invoiceNumber, items, grandTotal, balance, ... } BM-->>BC: Bill document saved alt Cashier session open BC->>CSM: Increment billCount, add to totalSales CSM-->>BC: Session updated end BC->>BC: commitTransaction() BC->>BM: Populate createdBy, handledBy, items.medicine BM-->>BC: Populated bill BC-->>BP: 201 { bill data, invoiceNumber } BP-->>Staff/Admin: Show bill summary + Print option

Sales Return Sales
Customer returns medicines from a previous bill. Stock is restored to inventory. Returns go through PENDING → APPROVED status.

Create Sales Return Flow

POST /api/sales-returns
sequenceDiagram actor Staff/Admin participant SRP as Sales Return Page participant SRC as SalesReturn Controller participant BM as Bill Model participant SRM as SalesReturn Model participant IM as Inventory Model Staff/Admin->>SRP: Select original bill & items to return Staff/Admin->>SRP: Enter return quantity & reason SRP->>SRC: POST /api/sales-returns { billId, items, reason, refundMode } SRC->>SRC: validateSalesReturnPayload() SRC->>SRC: Start MongoDB session & transaction SRC->>BM: findOne({ _id: billId, isDeleted: false }) BM-->>SRC: Original bill found SRC->>SRM: Aggregate already-returned quantities for this bill SRM-->>SRC: returnedQuantityMap loop For each return item SRC->>SRC: Find original bill item by index SRC->>SRC: Validate returnQty ≤ (originalQty - alreadyReturned) SRC->>IM: Find inventory batch by medicine + batchNumber IM-->>SRC: Batch found SRC->>SRC: Calculate refund amount (proportional) end SRC->>SRM: Create SalesReturn { bill, items, grandTotal, status: PENDING } SRM-->>SRC: SalesReturn created SRC->>SRC: commitTransaction() SRC-->>SRP: 201 { salesReturn data } SRP-->>Staff/Admin: Return created, pending approval Note over Staff/Admin,SRM: Admin approves the return Staff/Admin->>SRP: Click Approve on sales return SRP->>SRC: POST /api/sales-returns/:id/approve SRC->>SRC: Start transaction SRC->>SRM: findById(id) SRM-->>SRC: SalesReturn (status: PENDING) loop Restore inventory for each return item SRC->>IM: findOne({ medicine, batchNumber }) IM-->>SRC: Batch found SRC->>IM: Increment quantityAvailable by returnUnitQuantity IM-->>SRC: Stock restored end SRC->>SRM: Update status to APPROVED SRM-->>SRC: Saved SRC->>SRC: commitTransaction() SRC-->>SRP: 200 { approved return } SRP-->>Staff/Admin: Stock restored, refund processed
Create Purchase (Goods Receipt) Procurement
Records medicines received from a supplier. Each item is validated, GST is calculated, and inventory batches are created or updated.

Purchase Entry Flow

POST /api/purchases
sequenceDiagram actor Admin/Staff participant PP as Purchase Page participant PC as Purchase Controller participant SM as Supplier Model participant MM as Medicine Model participant HM as HSN Model participant PrM as Purchase Model participant IM as Inventory Model Admin/Staff->>PP: Select supplier, enter invoice number & date Admin/Staff->>PP: Add medicine items (batch, expiry, qty, MRP, rate, HSN) PP->>PC: POST /api/purchases { supplier, supplierInvoiceNumber, items, ... } PC->>SM: findOne({ _id: supplier, isDeleted: false }) SM-->>PC: Supplier exists and is active PC->>PC: Start MongoDB session & transaction loop For each purchase item PC->>MM: findById(item.medicine) MM-->>PC: Medicine found PC->>HM: findOne({ hsnCode }) — if HSN provided HM-->>PC: HSN & GST percent resolved PC->>PC: Validate batch number, expiry date PC->>PC: Calculate CGST/SGST/IGST, line total, discount end PC->>PC: Sum subtotal, totalGst, apply bill-level discount → grandTotal PC->>PrM: generatePurchaseNumber() → P000001 PrM-->>PC: Next purchase number PC->>PrM: Create Purchase { purchaseNumber, supplier, items, grandTotal, ... } PrM-->>PC: Purchase document saved loop For each purchase item (inventory update) PC->>IM: findOne({ medicine, batchNumber }) IM-->>PC: Batch found or not alt Batch exists PC->>IM: Update quantityPurchased, freeQuantity, quantityAvailable IM-->>PC: Batch updated else New batch PC->>IM: Create new Inventory { medicine, batchNumber, expiryDate, qty, supplier, purchase, ... } IM-->>PC: New inventory batch created end end PC->>PC: commitTransaction() PC->>PrM: Populate supplier, createdBy, items.medicine PrM-->>PC: Populated purchase PC-->>PP: 201 { purchase data } PP-->>Admin/Staff: Show purchase summary, updated stock

Automated Purchase Orders (Reorder Engine) Procurement
The system auto-generates draft POs when stock falls at or below the reorder level. Runs every 6 hours. Admin reviews, approves, and marks as sent.

Auto Reorder Check Flow

POST /api/purchase-orders/run-reorder (scheduled every 6h)
sequenceDiagram participant Scheduler as Server Scheduler (6h) participant RS as Reorder Service participant IM as Inventory Model participant MM as Medicine Model participant PrM as Purchase Model participant POM as PurchaseOrder Model participant NM as Notification Model Scheduler->>RS: checkAndCreateReorderPOs() RS->>IM: Aggregate total stock per medicine (ACTIVE batches) IM-->>RS: stockMap { medicineId → totalQty } RS->>MM: Find all ACTIVE medicines with reorderLevel > 0 MM-->>RS: Medicine list with reorderLevel & preferredSupplier RS->>RS: Filter: medicines where stock ≤ reorderLevel alt No medicines need reorder RS-->>Scheduler: { createdCount: 0 } else Low-stock medicines found RS->>POM: Find existing DRAFT POs (get medicine IDs already covered) POM-->>RS: alreadyInDraft set RS->>RS: Remove medicines already in a draft PO loop For each low-stock medicine (not in draft) RS->>MM: Check preferredSupplier alt preferredSupplier set and active RS->>RS: Use preferredSupplier else No preferred supplier RS->>IM: Find last purchase supplier from inventory IM-->>RS: lastSupplier end RS->>PrM: Get lastPurchasePrice for medicine PrM-->>RS: Last purchase price RS->>RS: Calculate requestedQty = max(2×reorderLevel - currentStock, reorderLevel) RS->>RS: Group medicine under supplier end loop For each supplier group RS->>POM: generatePONumber() → PO00001 POM-->>RS: PO number RS->>POM: Create PurchaseOrder { status: DRAFT, generatedBy: AUTO, items } POM-->>RS: Draft PO created RS->>NM: bulkWrite upsert LOW_STOCK notification NM-->>RS: Notification saved end RS-->>Scheduler: { createdCount: N, itemCount: M } end
